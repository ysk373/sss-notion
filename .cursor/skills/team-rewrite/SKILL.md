# チーム・リライト（Team Rewrite）

## 目的

公開中のブログ記事を **漏れなく把握**し、**経験豊富なウェブライター複数・信頼度の高いブロガー複数・ソフトウェアエンジニア複数・シニア営業職複数・凄腕マーケター複数** に相当する視点を **サブエージェント（Task ツールの並列起動）** で再現する。各視点からの診断とリライト方針を統合し、**AdSense のプログラムポリシーや品質期待に沿う本文・構成** へ近づける。審査通過や収益を保証するものではない。

## 横断リライト観点（全ロール共通・必須）

リライト案・監査では、次の **3 点を必ずチェック**し、問題があれば具体例つきで指摘・修正案に含める。

1. **口語からブログとして自然な文体へ**  
   話し言葉のままのモタつき、冗長なフィラー、口頭限定の言い回しを整理する。一方で **ブログとしての自然さ**（適度な一人称・読者への語りかけ・現場感）は残し、論文調やマニュアル一色に均さない。

2. **文字数は必要十分**  
   その記事の約束（読者の課題・結論・手順・根拠）を **説得力をもって満たす分量** があるかを見る。薄い場合は **追記すべき具体**（例・手順・比較・前提・失敗談）を明示する。逆に **水増し・繰り返しのみの冗長** は削る。

3. **AI が書いたときに出る「AI っぽさ」を必ず消す**  
   テンプレ臭い見出しや前置き、三面張りの機械的パターン、過度に丁寧で温度がない均一文、「〜することができます」「〜について説明します」の連発、抽象的な要約のだら積み、接続のない箇条書きの列挙などを **人間のブログのリズム** に書き換える。削除だけでなく **具体語・実例・理由・段落間のつながり** で置き換える。

## プロジェクト前提（必読）

- 記事本文の正は **Notion**。リポジトリ内に記事 Markdown を新規作成しない（`AGENTS.md`）。
- データベース ID: `2d82403f-bc5c-819f-a7f6-f1caee97d49f`。
- 投資・新 NISA 等は `src/constants/series-hubs.ts` の `isInvestmentPost()` と `src/components/InvestmentDisclaimer.astro` を崩さない。特定銘柄の勧誘表現は増やさない。
- サイト全体の申請前方針・チェックリストは `adsense-blog-council` と整合させる。ポリシー起点の論点は必要に応じて **Web 検索で Google 公式ヘルプ** を確認する。
- **`npm run build` は原則ユーザーが実行する。** ユーザーが明示しない限り、エージェントはビルドを試行しない（ローカル `.env` と MCP 認証が一致しない場合があり、試行はノイズになりやすい）。

### 監査・リライトから除外してよいページ（運用）

ユーザーが「後日非公開」「テンプレ検証用」などと指定した場合、次の slug は **インベントリに残っていても、深掘り監査・本文リライトの対象から外してよい**。`scripts/export-content-posts.mjs` は変更しない。

- `astro-notion-blog-introduction`
- `how-to-start-astro-notion-blog-setup`
- `supported-blocks`

## 開始時に親エージェントが確認すること

不足時のみ短く確認する。

- 対象は **公開済み（Published）記事のみ** か、下書きまで含むか。
- リライト結果の反映手段: Notion をユーザーが手で直す / Notion MCP で更新する、の希望。
- 1 回のセッションで扱う記事数の上限（全件を一度にリライト全文生成するとコンテキストを食うため **バッチ運用が既定**）。

## フェーズ 0: 全記事インベントリ

1. 環境変数が整っている場合、リポジトリで公開記事一覧を取得する。

   ```bash
   node scripts/export-content-posts.mjs
   ```

   出力は `tmp/content-posts.json`（`pageId`, `title`, `slug`, `tags`, `date`）。

2. 取れない場合は Notion MCP のデータベースクエリで同等の一覧を得る。固定ページ Slug（`about`, `contact`, `privacy-policy`, `disclaimer`）と `Info` タグ記事は、このスキルの **本文リライト対象から除外** してよい（`export-content-posts.mjs` と同様の扱い）。**後日非公開予定**などユーザー指定の slug があれば、上記 [監査・リライトから除外してよいページ（運用）](#監査リライトから除外してよいページ運用) に従い、対象から外す。

3. 親エージェントは一覧を **スプレッドシート代わりの表**（Markdown 表でよい）にし、総件数・タグ分布・日付レンジを 1 段落で要約する。

## フェーズ 1: サブエージェント・チームの組み方

**親エージェントは Task ツール（`subagent_type`: `generalPurpose` を推奨）を並列で複数起動**し、以下の **5 ロール × 最低 2 名分（合計 10 本以上の並列実行を目安）** の「チーム」を構成する。各サブエージェントには **同一インベントリ**、**このスキルの該当ロール定義**、および上記 **横断リライト観点（3 点）** をプロンプトに含める。名前は任意（例: ウェブライター A/B）。

並列数が多すぎる場合は、**(ア) コーパス全体の監査** と **(イ) 記事バッチごとのリライト** にフェーズを分け、(ア) でロール別に 2 本ずつ、(イ) でバッチ担当を割り当てる。

### ロール定義（サブエージェントへの指示にそのまま貼れる）

**1. 経験豊富なウェブライター**

- 見出し階層、リード、段落長、箇条書き、読み飛ばし耐性、結論ファースト。
- 冗長表現の削除、語調の統一、スキャナビリティ。
- **口語の整形**と **AI っぽいテンプレ表現の除去** を文体面で主担当とする。**分量**が約束に対して不足／過剰かも明示する。

**2. 信頼度の高いブロガー（E-E-A-T 重視）**

- 体験・検証・出典・限界の明示。煽り・誇大・未検証断定の除去。
- 「なぜこの記事を書いたか」「誰のどんな課題か」を本文でわかるようにする提案。
- **人間の熱量と現場感**がありつつ、信頼を損なう口語や曖昧さは **ブログとして自然な書き言葉** に直す。

**3. ソフトウェアエンジニア**

- 手順・コマンド・バージョン・前提環境の妥当性。誤った技術記述の指摘。
- コードや設定は最小・検証可能な形への整理（過剰な独自抽象は避ける）。
- 技術説明が **AI 生成っぽい抽象だけ** で終わっている場合は、**検証可能な具体**で厚みを出す案を出す。

**4. シニア営業職**

- 読者の業務課題との接続、ベネフィットの言語化、誤解しやすい点の先回り説明。
- 売り込み口調は避け、**読者の意思決定を助ける説明** に寄せる。
- **営業調・スライド調のカタカナ多用や決め台詞** が AI っぽさや軽さにつながる場合は、具体シーンベースの語りに落とす。

**5. 凄腕マーケター**

- タイトル・メタ的な訴求（クリックベイト禁止）、H2 の検索意図との対応、内部リンクのハブ案。
- `series-*` 目次や関連記事への誘導を乱さない範囲での改善。
- **キーワードスタッフィングやテンプレ H2 の罗列** を避け、検索意図に対して **必要十分な説明ボリューム** とセットで提案する。

### サブエージェント共通出力フォーマット

各サブエージェントは次の Markdown 構造で返す。**記事別所見** には可能なら **口語／分量／AI っぽさ** のどれが課題かを一文ずつ触れる。

```markdown
## ロール名（ペルソナ）

### 対象範囲

[割り当て記事 slug 一覧、または「全件俯瞰」の明示]

### 総評

[2〜5 文]

### 記事別所見

#### slug: xxx

- **現状の問題**: ...
- **リライト方針**: ...（口語・文字数・AI っぽさの該当あれば明示）
- **修正の優先度**: 高 / 中 / 低

### 横断リスク（任意）

[AdSense・信頼・誤情報・品質に関わる全体所見]
```

## フェーズ 2: 親エージェントによる統合

1. ロール間で矛盾する場合は **信頼性・ポリシー・事実正確性** を優先する。
2. 記事ごとに **優先度「高」だけを最初の実装キュー** に入れる。
3. `adsense-blog-council` の「共通評価基準」（独自性、薄さ、未完成ページ、禁止コンテンツ等）に照らし、**リライトで足りないものは「新規追記・検証タスク」として分離**する。
4. 統合時に **横断観点 3 点** がすべてカバーされているか確認し、抜けがあれば親が補完レビューを書く。

## フェーズ 3: リライト実行（バッチ）

1. **1 バッチ 3〜8 記事** を目安に、Notion から本文を取得できる手段（MCP、キャッシュ、`npm run cache:fetch` 後のデータ）で対象本文を読む。**利用可能なら Notion MCP**（`notion-fetch` で取得、`notion-update-page` で本文・プロパティ更新）による反映ができる。ユーザーが手作業を希望する場合は差分・全文案のみ渡す。
2. 実際の修正稿は **記事ごとに「変更サマリ」「見出し案」「全文または差分がわかる形」** で出力する。**リライト後に口語・分量・AI っぽさの自己チェック結果** を 1 行ずつ添える。投資記事は免責コンポーネント前提を維持し、免責本文を記事個別に増やしすぎない（サイト共通コンポーネントが正）。
3. サイトビルドの確認が必要なときはユーザーに依頼する。ユーザーが `npm run build`（や同等の検証）の実行を**明示した場合に限り**、エージェント側で試行してよい。Notion のみ変更でコードを触らない場合でも、ビルドは**既定では走らせない**。

## 最終デリバリー形式（親エージェントがユーザーへ）

```markdown
## チーム・リライト結果サマリ

- 対象件数: ...
- バッチ計画: ...

## 優先リライトキュー（高）

1. [title](slug) — 理由

## 記事別リライト案

### [title] (`slug`)

- **合意した変更方針**: ...（口語・分量・AI っぽさ）
- **修正稿または差分**: ...
- **リライト後セルフチェック**: 口語 / 分量 / AI っぽさ — OK または要追随
- **Notion 反映チェックリスト**: [ ]

## ポリシー・免責・固定ページ

- [サイト全体で確認したい点]

## 次のアクション

- [ユーザーまたは MCP での Notion 更新順]
```

## 既存スキルとの使い分け

- **運用方針・申請前監査・記事計画が主目的**: `adsense-blog-council`。
- **全記事を対象にしたリライト実行とマルチペルソナ監査が主目的**: このスキル（`team-rewrite`）。
- **新規テーマの発想**: `blog-idea-council`。

## 注意

- API トークンや `.env` をサブエージェントのプロンプトに貼らない。
- 技術記事で環境変数名（例: `ROS_LOCALHOST_ONLY`）が Notion→ブログの変換でリンク化・壊れとして出力されることがある。公開前に該当記事の HTML / MD を目視し、**変数名の表記崩れ**があれば Notion 側で修正する。
- 短編小説や `stories` 系は、リライト時も **文体・世界観を壊さない** ことをロール 2 に明示する。
