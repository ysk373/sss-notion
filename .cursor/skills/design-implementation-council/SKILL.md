# Design & Implementation Council

## 目的

単一視点での設計・実装判断を避け、複数の専門視点を持つサブエージェントで発散・批評・統合する。議論の進め方は **トヨタ流問題解決8ステップ** に沿い、ギャップの言語化から標準化まで一連で扱う。SSS Blog / Astro・Notion 連携リポジトリでは、`AGENTS.md` の方針（最小構成、秘密情報の非コミット、既存パターンへの寄せ）を前提にする。

## トヨタ流問題解決8ステップとの対応

Council の各フェーズで、次の8ステップを **順に埋める**。飛ばさず、データや根拠が弱いステップは明示的に「未確定」として残す。

| ステップ | 内容 | Council での扱い |
|---------|------|------------------|
| 1. 問題の明確化 | あるべき姿と現状のギャップを定義。対策ありきにしない | 開始時に親エージェントが1段落で整理。数値・再現手順・影響範囲があれば必ず書く |
| 2. 現状把握 | 事実ベースで現状を把握（コード、構成、制約）。問題のブレークダウン・層別で事実を切り分ける | 下記「ステップ2のルール」に従う。原因の深掘りはステップ4へ回す |
| 3. 目標設定 | 達成基準、スコープ、非目標を具体化 | 「完了の定義」「やらないこと」を明記 |
| 4. 要因解析（原因究明） | 真因・ボトルネック・制約の整理（なぜギャップが生じたか） | 下記「ステップ4のルール」に従い、なぜなぜ分析で「なぜ」を最低5回問う。複数視点で要因を列挙し優先度付け |

### ステップ2（現状把握）のルール

- 問題の**ブレークダウン**を行い、**層別**（観点・単位・場所などでたてわける）して整理する。
- **事実の切り分け**が目的とする。観測できる事象・データ・再現条件・影響範囲を「わかった／わからない」で分ける。
- **原因の深掘りはしない**。「なぜ」の連鎖や真因の断定はステップ4に任せ、ステップ2では混ぜない。

### ステップ4（要因解析・原因究明）のルール

- **なぜなぜ分析**を行い、**「なぜ」を最低5回**問うこと（5 Whys の回数条件を満たすまで掘り下げる）。
- 各「なぜ」の回答は事実と推測を分け、推測は検証方法または保留理由を添える。
| 5. 対策立案 | 対策案の比較、採用案と代替案 | 設計案・実装方針・ロールバック案を含める |
| 6. 対策実施 | 採用した対策の実行 | Council の出力は「実施手順・チェックリスト・タスク分割」。実際の編集は別タスクで実行してよい |
| 7. 効果確認 | ギャップが解消したかの検証 | ビルド、テスト、手動確認項目、`npm run build` 等のプロジェクト既定 |
| 8. 標準化・再発防止 | ルール・ドキュメント・テンプレへの反映、横展開 | `AGENTS.md`、既存スキル、コメント、チーム内共有事項の更新提案 |

## 開始時に確認すること

ユーザーの依頼が曖昧な場合は、先に次を短く確認する。

- 変えたいこと・変えたくないこと（スコープと非目標）
- 背景・きっかけ（バグ、性能、保守性、新機能、審査対応など）
- 期限・リスク許容度（互換性破壊の可否、段階リリースの要否）
- 既存の関連Issue・PR・Notionページの有無

明確な指定がなくても進める場合は、リポジトリの `AGENTS.md` と既存コードパターンを正として仮説を置く。

## 基本ワークフロー

1. **ステップ1〜3（親エージェント）**: 問題の明確化、現状把握（ブレークダウン・層別・事実の切り分けのみ）のインプット整理、目標設定を1つのブリーフにまとめる。
2. **ステップ4〜5（並列サブエージェント）**: 下記の各視点でサブエージェントを起動し、要因解析と対策・設計の提案を出させる。
3. **統合（親エージェント）**: 矛盾・重複を解き、採用案と代替案、オープンイシューを整理する。
4. **第2ラウンド（任意）**: 統合案に対し、「批評・リスク」視点のサブエージェントで再レビューする。
5. **ステップ6〜8（親エージェント）**: 実施順序、検証方法、標準化・再発防止案を最終パッケージに含める。

外部仕様やセキュリティ境界が絡む場合は Web 検索や公式ドキュメントで事実確認する。

## サブエージェント構成

### アーキテクチャ・設計視点

役割: モジュール境界、データフロー、拡張性、既存パターンとの整合を評価する。

出力:

- 推奨構成案（案が複数ある場合は A/B）
- インターフェースと責務の切り方
- 将来変更が入りやすい箇所と安定させる箇所
- 既存コードとの整合・移行パス

### 実装・保守性視点

役割: 変更規模、可読性、命名、型安全、デバッグ容易性を評価する。

出力:

- 具体的な変更単位の提案（ファイル・関数単位でよい）
- 既存スタイルとの一致点・例外が必要な点
- 技術的負債を増やさないための最小差分の指針

### テスト・検証視点

役割: 回帰リスク、確認項目、自動テストの要否を整理する。

出力:

- 必須の手動確認シナリオ
- 追加・更新すべきテストの有無
- `npm run build` 以外に必要な検証

### 運用・互換・デプロイ視点

役割: SSG、環境変数、キャッシュ、Notion/R2、GitHub Pages 等の運用への影響を見る。

出力:

- デプロイ前後のチェック
- 既存URL・キャッシュ・ビルドパイプラインへの影響
- ロールバックの考え方

### セキュリティ・プライバシー視点

役割: 秘密情報の取り扱い、入力検証、公開範囲、依存関係のリスクを確認する。

出力:

- ハードコードやログに関する注意
- 公開してはいけない情報の経路
- 依存追加時の検討点

### 批評・デビルズ・アドボケイト視点

役割: 楽観バイアスを潰す。見落とし、過小評価したリスク、代替案の再提示を行う。

出力:

- 統合案に対する反論・弱点
- 取りこぼしたエッジケース
- 「やらない」選択の正当化が必要な点

## サブエージェントへの共通指示

各サブエージェントには、次の形式で返させる。ステップ番号を見出しに含め、**推測には「推測」と明記**する。

```markdown
## 視点名

### ステップ4 要因解析（原因究明）

- [なぜなぜ分析: 「なぜ」を最低5回。各段の事実／推測を区別]
- [真因・制約・前提の整理。優先度付け]

### ステップ5 対策・設計案

- [提案。代替案があれば列挙]

### リスク・未確定事項

- [確認が必要な点]

### ステップ7 効果確認に必要なこと

- [検証項目]
```

## 統合時の評価基準

親エージェントは次の観点で採点・優先付けする。

- ステップ1のギャップに直接効くか（目的から逸脱していないか）
- ステップ3の目標を満たすか（非目標を侵していないか）
- 変更が最小で済むか（YAGNI、過度な抽象化を避ける）
- `AGENTS.md`・既存コード規約に沿うか
- 秘密情報や運用リスクが増えないか
- ステップ7で検証可能か、ステップ8でルール化できるか

判断が割れた場合は、**本番データ・ユーザー安全・保守性**を優先し、次いでスコープ縮小で合意を取る。

## 最終出力フォーマット

```markdown
## 結論サマリ

[採用する方針を2〜5文]

## トヨタ流8ステップ整理

### 1. 問題の明確化（ギャップ）

### 2. 現状把握（ブレークダウン・層別・事実の切り分け／原因の深掘りはしない）

### 3. 目標設定（完了の定義・非目標）

### 4. 要因解析・原因究明（なぜ×最低5回のなぜなぜ分析を含む）

### 5. 対策立案（採用案・主要代替案）

### 6. 対策実施（タスク分割・順序）

### 7. 効果確認（検証リスト）

### 8. 標準化・再発防止（ドキュメント・ルール案）

## オープンイシュー

- [未決定事項と次のアクション]

## エージェント間の主な争点と解消方針

- [争点]: [解消方針]

## 次のアクション

- [最初に着手する具体タスク]
```

## 既存スキルとの使い分け

- ブログ記事のアイディア・編集会議が主目的なら `blog-idea-council` を使う。
- AdSense・収益化・審査前のブログ運用が主目的なら `adsense-blog-council` を使う。
- **コードベースの設計・実装変更の合議レビュー**が主目的ならこの Skill を使う。

## プロジェクト固有メモ

- コード変更後の確認は可能な範囲で `npm run build` を実行する（環境変数不足などで失敗する場合は理由を記録する）。
- Notion を記事本文の正とする方針は変えない。設計メモをリポジトリに増やしすぎない（ユーザーが明示した場合を除く）。
- シリーズ目次・投資免責などサイト固有ルールは `AGENTS.md` および `src/constants/series-hubs.ts` 等を参照する。
