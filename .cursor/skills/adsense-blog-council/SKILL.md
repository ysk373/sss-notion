# AdSense Blog Council

## 目的

SSS BlogをGoogle AdSenseに登録できる状態へ近づけるため、複数視点のサブエージェントでブログ運用方針を議論し、申請前に必要な修正、記事計画、継続運用ルールを決める。審査通過を保証するものではなく、公式ポリシーに沿ってリスクを減らすことを目的にする。

## 開始時に確認すること

不足している場合は、先に次を短く確認する。

- 申請予定のURL
- 現在の公開記事数と主なカテゴリ
- 既に用意済みの固定ページ: プロフィール、問い合わせ、プライバシーポリシー、免責事項
- 優先したいテーマ領域: 技術、投資、日記、学習記録、短編小説
- 今回の目的: 審査前監査、運用方針策定、記事計画、既存記事改善

明確な指定がない場合は、SSS Blogの既定読者として「組み込みシステムエンジニア、DSP開発者、自動車業界技術者、投資に興味を持つ会社員・主婦」を想定する。

## 基本ワークフロー

1. 親エージェントが目的、対象URL、現状、評価基準を1段落で整理する。
2. AdSense公式ポリシーや最新要件が関係する場合は、Web検索でGoogle公式ヘルプを確認する。
3. 次のサブエージェントを並列起動し、各視点で提案とリスクを出す。
4. 親エージェントが意見を統合し、矛盾、重複、優先順位を整理する。
5. 必要に応じて第2ラウンドとして、批評・改善エージェントに結論案をレビューさせる。
6. 最終的に「申請前に必須」「申請前に推奨」「申請後に継続改善」に分類する。

## サブエージェント構成

### AdSense審査・ポリシー視点

役割: Google Publisher Policies、AdSense Program policies、価値の低い広告枠、独自性不足、未完成ページ、禁止・制限コンテンツのリスクを確認する。

出力:
- 審査上の懸念
- ポリシー確認が必要なページや記事
- 申請前に直すべき不足
- 公式情報で確認すべき項目
- 審査通過可能性を下げる表現や構成

### 編集長・ブランド視点

役割: ブログ全体のテーマ一貫性、読者価値、運営者の専門性、SSS Blogらしさを評価する。

出力:
- ブログの主軸テーマ案
- 読者に約束する価値
- 残すカテゴリ、弱めるカテゴリ
- プロフィールで訴求すべき経験
- AdSense申請前に見せたい代表記事

### SEO・検索需要視点

役割: 検索意図、ロングテールキーワード、競合との差別化、内部リンク設計を検討する。

出力:
- 優先キーワード群
- 検索意図ごとの記事案
- 競合に勝つ差別化要素
- 内部リンクでつなぐ記事クラスター
- タイトルと見出しの改善方針

### 読者体験・信頼性視点

役割: 初見読者が安心して読めるか、運営者情報、問い合わせ導線、プライバシーポリシー、ナビゲーション、スマホ表示を確認する。

出力:
- 必要な固定ページ
- ヘッダー、フッター、カテゴリ導線の改善案
- 読者が迷う箇所
- 信頼性を上げるプロフィール要素
- 申請前チェック項目

### 記事品質・独自性視点

役割: 薄い記事、一般論だけの記事、AI生成感の強い記事、引用過多、重複記事を見つけ、体験や検証で補強する。

出力:
- 改善対象の記事タイプ
- 追記すべき体験、検証、図解、コード、比較表
- 統合・非公開を検討する記事
- 1記事あたりの品質基準
- リライト優先順位

### 運用計画視点

役割: 申請までの現実的なスケジュール、記事公開順、メンテナンス、審査後の継続運用ルールを決める。

出力:
- 申請までのタスク順
- 2週間から1か月の公開計画
- 申請前の最終確認日
- 審査後の更新ルール
- Notionで管理すべきステータスやプロパティ

## 共通評価基準

各サブエージェントは、次の観点で判断する。

- 独自の経験、検証、観察が含まれている
- 読者の課題を具体的に解決している
- 未完成、空ページ、極端に薄いページがない
- コピー、転載、引用依存になっていない
- 禁止コンテンツや誤解を招く表現がない
- 運営者、問い合わせ、プライバシー関連の信頼性がある
- スマホで読みやすく、広告なしでも価値がある
- 申請前に代表記事として見せられる品質になっている

## サブエージェントへの共通指示

各サブエージェントには、次の形式で返させる。

```markdown
## 視点名

### 結論
[この視点での最重要判断を1から3文]

### 良い点
- [評価できる点]

### リスク
- [審査、読者、運用上のリスク]

### 必須対応
- [申請前に必ず対応すること]

### 推奨対応
- [できれば対応すること]

### 記事・ページ案
- [必要な記事または固定ページ]
```

## 統合時の判断

親エージェントは、提案を次の3段階に分類する。

- 申請前に必須: 審査や信頼性に大きく影響する不足。固定ページ、未完成ページ、薄すぎる記事、ポリシーリスクなど。
- 申請前に推奨: 通過可能性や読者体験を上げる改善。代表記事の強化、内部リンク、カテゴリ整理など。
- 申請後に継続改善: 収益化後も育てる施策。記事クラスター拡張、SEO改善、リライト運用など。

判断が割れた場合は、AdSense審査・ポリシー視点と読者体験・信頼性視点を優先する。短期的なSEOより、独自性、信頼性、完成度を優先する。

## Notion連携

記事案を下書き化する場合は、`create-blog-post` SkillまたはNotion MCPを使う。本文作成前に、タイトル、Slug、Tags、Excerpt、アウトラインをユーザーに確認する。

SSS BlogのNotionデータベースIDは `2d82403f-bc5c-819f-a7f6-f1caee97d49f`。タグ候補は既存運用に合わせ、`Diary`、`How-to`、`Introduction`、`Document` を優先する。

## 既存スキルとの使い分け

- 記事テーマの発散が主目的なら `blog-idea-council` を使う。
- AdSense申請、収益化、審査前監査、ブログ運用方針が主目的ならこの Skill を使う。
- 記事案が確定したら `create-blog-post`、サムネイルが必要なら `add-thumbnail` を使う。
